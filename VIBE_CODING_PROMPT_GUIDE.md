# Prompt Guide

这份指南的目标很简单：**让 LLM 更稳定、更短路径、更少跑偏。**

适用场景：

- 你要它先分析，不要直接改
- 你要它先给方案，再决定是否实现
- 你要它做复杂任务，但不想被无意义追问打断
- 你要它少做全仓调研、少擅自扩 scope、少补脑

---

## 一、好 Prompt 的 6 个组成

一条高质量需求，尽量包含这 6 件事：

1. **目标**：最终想得到什么
2. **范围**：允许改什么，不允许改什么
3. **阶段**：现在要分析、设计、实现、验证，还是只做其中一段
4. **约束**：最小改动、不要新增依赖、不要改文件、不要重型调研等
5. **决策边界**：哪些事可以自主处理，哪些事必须先确认
6. **验证方式**：要跑什么、证明什么算完成

如果只说“帮我优化一下”“你看着办”，模型通常会：

- 自己补需求
- 自己补范围
- 自己补成功标准
- 然后越做越多

示例：

### 较差写法

```text
帮我优化一下登录逻辑，你看着改。
```

### 更好写法

```text
先不要修改任何文件。

请分析当前登录逻辑的问题，重点判断是输入校验、错误提示还是状态管理导致体验差。

要求：
- 只做最小必要阅读
- 只看登录入口和直接相关的 1-2 个文件
- 给出问题判断、影响范围、建议下一步
- 完成本轮分析后先停住
```

## 二、最常用的 4 类 Prompt 模板

### 1. 分析类：先看，不动

适合：先了解问题、先找原因、先做判断。

模板：

```text
先不要修改任何文件。

请基于当前项目做最小必要分析，回答：
1. 问题是什么
2. 影响范围是什么
3. 你建议的下一步是什么

要求：
- 只做最小必要阅读
- 不要直接进入实现
- 完成本轮分析后先停住
```

示例：

```text
先不要修改任何文件。

请分析订单提交接口为什么偶发返回 500。

要求：
- 只查看接口入口、直接调用链和相关错误处理
- 不要修改代码
- 给出最可能原因、影响范围、建议下一步
- 完成本轮分析后先停住
```

### 2. 方案类：先比较，再决定

适合：有设计分歧、要比较方案、还没决定方向。

模板：

```text
先不要修改任何文件。

请基于当前项目做最小必要分析，然后：
1. 给我 2 个可行方案
2. 分别说明：核心思路、优点、风险、适用场景
3. 给出你的推荐方案和理由
4. 如果这里存在关键决策点，请明确列出来，等我确认

要求：
- 只做最小必要阅读
- 不要直接进入实现
- 完成本轮分析和一个关键确认点后先停住
```

示例：

```text
先不要修改任何文件。

请评估我们后端缓存策略应该继续放在 API 层，还是下沉到 service 层。

请先：
1. 基于当前实现做最小必要分析
2. 给出两个可行方案
3. 说明核心思路、优点、风险、适用场景
4. 给出你的推荐方案
5. 如果这里有关键决策点，请明确列出来，等我确认

不要直接进入实现。完成本轮分析和一个关键确认点后先停住。
```

### 3. 实现类：单点小改

适合：单文件、小改动、低风险。

模板：

```text
请修改 <文件路径> 中的 <具体问题>。

要求：
- 只允许修改这一个文件
- 做最小改动
- 不要新增文件
- 不要做全仓调研
- 改完后做最小验证并汇报
```

示例：

```text
请修改 `src/components/LoginButton.tsx` 中的按钮文案。

要求：
- 只允许修改这一个文件
- 将 “Submit” 改为 “Sign in”
- 不要新增文件
- 不要做全仓调研
- 改完后做最小验证并汇报
```

### 4. 复杂任务类：先立边界，再开工

适合：多文件、跨模块、可能多轮修复。

模板：

```text
请先不要修改任何文件。

先基于真实实现判断这件事应该如何推进，然后给出：
- Goal
- Scope
- Non-goals
- Rubric
- Budget
- Stop conditions

再说明：
- 哪些步骤可以并行，哪些必须串行
- 哪些分歧你可以自主处理
- 哪些分歧必须先问我

在我批准前，不要开始修改。
```

示例：

```text
请先不要修改任何文件。

我想统一项目里认证、会话续期、权限校验这三块逻辑的边界。

请先基于真实实现判断这件事应该如何推进，然后给出：
- Goal
- Scope
- Non-goals
- Rubric
- Budget
- Stop conditions

再说明：
- 哪些步骤可以并行，哪些必须串行
- 哪些分歧你可以自主处理
- 哪些分歧必须先问我

在我批准前，不要开始修改。
```

## 三、怎么让它“少调查一点”

如果你不写，模型经常会为了显得谨慎而过度调研。

建议直接加这句：

```text
请避免为了“显得谨慎”而做无边界仓库级探索；只收集足以支持本轮判断的最小证据。
```

如果你只想要一轮分析，不想被拉成长流程，再加这句：

```text
完成本轮目标后先停住，不要自动扩展到下一阶段。
```

---

## 四、怎么让它“少问几轮”

有些模型已经学会了先确认，但会把一个问题拆成 3~5 轮追问，导致会话很长。

建议你明确写：

```text
本轮只允许发起一个最高优先级确认点；其余开放问题先归纳，不要连续多轮追问。
```

示例：

```text
如果有多个开放问题，本轮只提出最影响结果的那一个；其余问题先归纳为“待后续确认”，不要连续多轮追问。
```

## 五、哪些能控，哪些不能控

这是最容易误判的一点。

### 不能直接精确控制的

- 模型内部到底想了多少
- 隐藏推理到底有多长
- 某些平台展示出来的思考文本长度

这些更接近模型或平台行为，不是 Prompt 能完全硬控的。

### 可以明显收紧的

- 它读多少文件
- 它搜多深
- 它本轮最多问几个问题
- 它什么时候必须停住
- 它是先给结论，还是先做半轮研究
- 它会不会把分析题自动扩成设计题、实现题、验证题

所以真正应该约束的不是“别想太多”，而是：

- **只收集最小必要证据**
- **只发起最高优先级确认**
- **完成本轮目标后立即停住**
- **不要自动进入下一阶段**

## 六、怎么给反馈，LLM 最容易改对

最有效的反馈，不是“感觉不对”，而是下面这 5 段式：

1. **保留什么**：哪些做得对，后续继续保持
2. **改掉什么**：哪一段行为不对
3. **原因是什么**：为什么这是问题
4. **以后怎么做**：替代行为
5. **下一步是什么**：现在继续做什么，不做什么

模板：

```text
这次有 2 点做得对：
- ...
- ...

但有 3 点要改：
- ...
- ...
- ...

原因：
- ...

以后请改成：
- ...

现在先做：
- ...
先不要做：
- ...
```

示例：

```text
这次有 2 点做得对：
- 你先确认了阶段，没有直接改代码
- 你给出了明确的下一步建议

但有 3 点要改：
- 读取范围过大
- 一个确认点拆成了多轮追问
- 分析后没有及时停住

原因：
- 这轮目标只是判断方向，不是展开完整实现

以后请改成：
- 只收集支持本轮判断的最小证据
- 只提出一个最高优先级确认点
- 完成本轮目标后立即停住

现在先做：
- 重新给我一版最小证据下的判断和单一确认点
先不要做：
- 不要继续扩展到实现计划
```

## 七、提需求时最容易漏掉的 4 件事

### 1. 忘了说“现在是哪个阶段”

比如你只是想先分析，但没写“先不要改文件”，模型就可能直接上实现。

### 2. 忘了说“完成后要不要停住”

如果不写，模型经常会把“分析完”自动扩展成“继续设计、继续计划、继续问下一轮”。

### 3. 忘了说“可自主继续”和“必须确认”的边界

这会导致两种坏结果：

- 要么动不动就停机
- 要么把该确认的东西也直接越过去

### 4. 忘了说“验证强度”

小改动如果不写，模型会为了保险跑很多不必要检查。

## 八、通用提示词补丁

如果你希望大多数任务都更克制、更稳，可以常驻加下面几句：

```text
- 只做最小必要阅读，不要一上来做重型全仓调研
- 如果存在关键决策点，请明确列出，等我确认
- 完成本轮分析与一个关键确认点后先停住
- 不要为了显得谨慎而自动扩展成下一阶段
- 对可逆、低风险、范围内的分歧可自主继续；高影响决策先确认
```

示例：

```text
请默认采用下面这些执行约束：
- 只做最小必要阅读，不要一上来做重型全仓调研
- 如果存在关键决策点，请明确列出，等我确认
- 完成本轮分析与一个关键确认点后先停住
- 不要为了显得谨慎而自动扩展成下一阶段
- 对可逆、低风险、范围内的分歧可自主继续；高影响决策先确认
```

---

## 九、一句话原则

**好的 Prompt 不是“写很多字”，而是把：目标、边界、阶段、决策、验证，说清楚。**

这样 LLM 才能：

- 少猜
- 少绕
- 少扩 scope
- 少无效调查
- 在该停时停，在该继续时继续
